# Requirements — 人文・社会科学 読書ロードマップ（reading-tree）
status: approved
updated: 2026-07-04
based_on: .dev/business-plan.md (status: approved), .dev/marketing-plan.md (status: approved)

## 目的
人文・社会科学系分野（まずは経営学）を独学・研究で学ぶ人が、「何から読めばいいか」「どの順番で読み進めればいいか」を、本同士の依存関係とレベル（初心者→専門）を可視化したツリー/グラフで把握できるようにする。

## 対象ユーザー
- 一次ユーザー: 開発者本人（商学研究科・経営学専攻）
- 将来的な二次ユーザー: 同分野を独学・研究する学生・社会人（ゼミ・Xの学術クラスタ経由で獲得予定）

## MVPスコープ（機能要件）
1. **ツリー/グラフ表示**: 分野（初期は経営学のみ）ごとに、本をノードとし、「前提→次に読む本」の依存関係と「初心者→中級者→専門」のレベルを可視化する。UIイメージはObsidianのgraph view / Connected Papersに近い、ノードとエッジで構成されたインタラクティブな図
2. **初期コンテンツ投入**: 経営学分野のツリーを開発者本人が手動で作成し、シードコンテンツとする。書籍データソースは以下の2系統を統合する:
   - books.csv（444冊、経営学78冊など）— 開発者本人の既読・収集書籍
   - 日本の大学が公開しているシラバスから収集する経営学・MBA関連書籍リスト。対象校: 東京大学、京都大学、一橋大学、早稲田大学、慶應義塾大学、青山学院大学、東京理科大学、立教大学、明治大学、および他に公開シラバスを持つ日本の主要大学。学士・修士・博士のすべての課程を対象に検索する
3. **本の追加提案**: ログイン済みユーザーが「この分野にこの本を追加/この依存関係を追加」を提案できる（GitHubのPRに近いフロー）
4. **レビュー・承認フロー**: 提案は「下書き→レビュー中→承認/却下」のステータスを持つ。MVPでは開発者本人のみがレビュー権限を持つ（専門家自動マッチングは非スコープ）
5. **認証**: Googleログイン（OAuth）のみ。提案には投稿者を紐付ける
6. **アフィリエイト表示**: 各本ノードにAmazon/楽天の紹介リンクを表示し、「このリンクは運営費に充てています」という趣旨の透明な表記を添える
7. **LP（ランディングページ）**: marketing-plan.mdのLP要件に基づき、トップページを以下のセクション構成で実装する。Hero → デモ（ツリーの埋め込み） → 共感（課題描写） → 機能（欲求ベース3点） → 比較表 → Pricing → testimonial → フッターCTA。コピーはmarketing-plan.mdの実文をそのまま使用する（意訳・省略しない）
8. **計測イベント**: PostHogで以下のイベントを送出する: `page_view`, `cta_click_hero`, `cta_click_footer`, `tree_node_click`, `proposal_submitted`, `proposal_approved`

## 非スコープ（やらないこと）
- CS・プログラミング分野のツリー（roadmap.sh等の既存解と競合するため対象外）
- 読了ステータス管理・感想・評価などの汎用読書管理機能
- 専門家への自動レビュー依頼・マッチング機能（ユーザー数が増えるまで非対応）
- 完全オープンなwiki編集（誰でも即時反映は不可。必ず提案→承認を経る）
- 広告バナーの掲載
- メール+パスワード認証（OAuthのみで十分とする）

## 非機能要件
- **ホスティング予算**: 無料枠内（$0）を優先する構成とする
- **開発ペース**: 週数時間規模での個人開発。MVPは最小構成でまず経営学1分野・手動投入コンテンツのみで公開できることを優先する
- **想定規模**: MVP時点で同時アクセス数十人程度を想定（大規模スケールは考慮しない）
- **データ**: books.csv（444冊）に加え、日本の大学（東京大学・京都大学・一橋大学・早稲田大学・慶應義塾大学・青山学院大学・東京理科大学・立教大学・明治大学ほか公開シラバスを持つ主要大学、学士・修士・博士課程横断）の経営学・MBA関連シラバスから収集した書籍リストを統合し、初期データソースとして使う

## セキュリティ要件
- 認証はOAuth（Google）のみとし、パスワードを自前で保持・管理しない
- 提案フォームなどユーザー入力はサーバー側でバリデーション・サニタイズし、XSS/インジェクションを防ぐ
- レビュー権限（承認・却下操作）は開発者本人のアカウントに限定し、権限チェックをサーバー側で行う
- OAuthクライアントシークレット・DB接続情報などは環境変数で管理し、リポジトリにコミットしない
- アフィリエイトタグ・リンクの改ざん防止（提案経由で任意リンクを混入させない。書籍リンクはISBN等の識別子からアフィリエイトタグ付きURLをサーバー側で生成する）
