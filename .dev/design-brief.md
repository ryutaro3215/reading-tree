## プロンプト

### 1. LP（トップページ）

```
reading-treeのランディングページを作成してください。

- Goal: 若手ビジネスパーソン・コンサルタントに対し、経営学を体系立てて学べる読書ツリーの存在を伝え、無料でツリーを見てもらうこと（marketing-plan.md 主戦場: X経由の流入を想定）

- Audience: 経営学の基礎知識を求める若手ビジネスパーソン・コンサルタント（例: コンサル1年目）。フレームワークは知っているが、その土台となる経営学の体系的知識に不安がある層。デバイス: mobile / desktop 両対応（Xからのスマホ流入を優先）

- Layout（上から順に）:
  1. Hero
  2. デモ（ツリー/グラフの埋め込みイメージ）
  3. 共感（課題描写）
  4. 機能（欲求ベース3点）
  5. 比較表
  6. Pricing
  7. testimonial
  8. フッターCTA

- Content（実文をそのまま使用。要約・言い換え禁止）:
  1. Hero
     - 見出し: 「コンサル1年目の経営学、遠回りゼロで読む順番がわかる」
     - サブコピー: 「フレームワークは知っていても、その土台がない人へ。経営学78冊と主要大学MBAのシラバスから、初級→中級→専門の順に読むツリーを無料で公開しています。」
     - CTAボタン文言: 「経営学のツリーを見る」
     - Pricing情報をヘッダー直下にも一文で表示: 「完全無料。運営費はAmazon/楽天の紹介リンクでまかなっています」
  2. デモ
     - 経営学分野のツリー/グラフを操作できるプレースホルダー（ノードとエッジで構成された図。実データは後でdev-builderが接続）。説明文より先に配置すること
  3. 共感（課題描写）
     - 本文: 「『世界標準の経営理論を読め』と言われても、その前に何を読むべきかは誰も教えてくれない。会議で先輩が当たり前のように使うフレームワークの背景を、自分だけ知らない気がする。」
  4. 機能（欲求ベース、3カード）
     - カード1（時間）: 「次に読む本を選ぶのに毎回30分溶かす時間をなくす」
     - カード2（地位）: 「会議で経営理論の土台を語れるようになる」
     - カード3（苦痛除去）: 「レベルの合わない本を買って挫折するリスクを減らす」
  5. 比較表（4列: 本サービス／おすすめ本10選ブログ／Notion・Obsidianで自作／何もしない。行: 読む順番の可視化／レベル別構成／費用／更新・拡充）
     - 本サービス: ○（ツリーで依存関係表示）／○／無料／○（提案→承認で継続更新）
     - おすすめ本10選ブログ: ×（羅列のみ）／×／無料／×（公開後は放置されがち）
     - Notion/Obsidianで自作: △（自分で作る手間）／△／無料（構築の手間コスト大）／△（自分次第）
     - 何もしない: ×／×／0／-
  6. Pricing
     - 一文: 「完全無料。運営費はAmazon/楽天の紹介リンクでまかなっています」（ヘッダー直下と重複表示でよい）
  7. testimonial
     - MVP時点ではプレースホルダー（「準備中」の表示、または枠のみ）。本文の創作はしない
  8. フッターCTA
     - 見出し: 「今日から、経営学の迷子をやめる」
     - CTAボタン文言: 「経営学のツリーを見る」（Heroと同一文言・同一導線）

- 制約: 配色3色（背景 #FFFFFF / 文字 #1A2233 / アクセント #1E3A8A）/ CTAは「経営学のツリーを見る」の1種類のみに統一（Hero・フッター含め全て同一文言）/ Pricing情報をヘッダー直下に掲載 / 1画面1メッセージを意識し各セクションの主張を1つに絞る / OG画像はツリーのグラフビジュアルをサムネイル品質でイメージ提案する
```

### 2. ツリー/グラフ表示ページ（経営学分野）

```
reading-treeの「ツリー/グラフ表示ページ」（経営学分野）を作成してください。

- Goal: 経営学分野の本を、依存関係（前提となる本→次に読む本）とレベル（初級→中級→専門）を持つノード・エッジのグラフとして可視化し、ユーザーが次に読む1冊を直感的に判断できるようにすること（requirements.md 機能要件1）。LPのデモセクションに埋め込む見た目のリファレンスも兼ねる

- Audience: LPのHeroから遷移してきた、経営学の基礎知識を求める若手ビジネスパーソン・コンサルタント。将来的には本の追加提案をするログイン済みユーザーも含む。デバイス: desktopでの閲覧・操作を主、mobileでも最低限閲覧できることを想定

- Layout（上から順に）:
  1. ヘッダー: 分野名「経営学」+ 1行説明 + トップページへ戻る導線
  2. メイングラフエリア: ノード（書籍）とエッジ（依存関係）で構成された、Obsidianのgraph viewやConnected Papersのような有機的な力学配置（force-directed layout）の図。画面の大部分を占める
  3. ノード詳細パネル: ノードクリック時にサイドパネルまたはモーダルで表示。書籍タイトル・著者・レベル（初級/中級/専門）・購入リンク（アフィリエイト）・「この分野に本を追加提案する」への導線
  4. フローティングまたはヘッダー内アクション: 「本を追加提案する」ボタン（1つのみ）

- Content（実文。requirements.md由来の値はそのまま使用、書籍データ自体はダミーでよい）:
  - 分野名: 「経営学」
  - レベルラベル: 「初級」「中級」「専門」（この3語をそのまま使用。言い換え禁止）
  - ノード詳細パネルの項目ラベル: 「著者」「レベル」「この本を購入する」「この分野に本を追加提案する」
  - サンプルノード（デモ用ダミーデータ、3〜5冊で可）: 「経営学入門」→「ゼミナール経営学入門」→「世界標準の経営理論」のような初級→中級→専門の連鎖を1本示す

- 制約: 配色はLPと同じ3色（背景 #FFFFFF / 文字 #1A2233 / アクセント #1E3A8A）/ レベルによってノードの色の濃淡またはサイズを変え、初級→専門が視覚的に分かるようにする / アクション（CTA相当）は「本を追加提案する」の1つのみに絞る / 1画面1メッセージを意識し、グラフ以外の要素で画面を埋めすぎない
```

## イテレーション用の修正プロンプト集
- CTAが2種類以上ある → 「CTAを『経営学のツリーを見る』1つに統合して」
- Pricing情報がヘッダーに見当たらない → 「ヘッダー直下に『完全無料。運営費はAmazon/楽天の紹介リンクでまかなっています』を追加して」
- 配色が3色を超えている → 「配色を背景#FFFFFF・文字#1A2233・アクセント#1E3A8Aの3色のみに戻して」
- testimonialセクションに架空の口コミ文が生成された → 「testimonialは準備中のプレースホルダー表示にして、文章は創作しないで」
- 比較表の項目が変わっている → 「比較表の行・列・○×△の内容を design-brief.md の通りに戻して」
- 見出し・サブコピー・CTA文言が言い換えられている → 「コピーはdesign-brief.mdの実文と一字一句同じに戻して」
- グラフが格子状・整列レイアウトになっている → 「Obsidianのgraph viewやConnected Papersのような有機的な力学配置（force-directed layout）にして」
- ノードのレベル（初級/中級/専門）が視覚的に区別できない → 「レベルごとにノードの色の濃淡またはサイズを変えて、初級→専門の階層が一目でわかるようにして」
- グラフ画面にCTAやボタンが複数ある → 「アクションを『本を追加提案する』の1つだけに絞って」

## Claude Design での手順
1. claude.ai/design で新規プロジェクト作成
2. 本ブリーフの「デザイントークン」を最初に貼って宣言する
3. 上記プロンプトを1画面ずつ貼る（LP→ツリー/グラフ表示ページの順。一度に両方貼らない）
4. 修正はインラインコメントで具体的に（「ここを8px詰める」等）。崩れた場合は上記の修正プロンプト集を使う
5. デザインが固まったら「Export」→「Hand off to Claude Code」を選択する。Claude Designがコンポーネント構造・デザイントークン・レイアウト階層・参照アセットを含むハンドオフバンドルを生成し、それを取り込むためのプロンプト（バンドルのURL付き）が提示される
6. （ハンドオフ機能が使えない環境向けのフォールバック）: standalone HTML でエクスポートし、プロジェクトの `design/` 配下に配置する（例: `design/lp.html`）

## dev-builder への引き継ぎ
- 主経路: Claude Designの「Hand off to Claude Code」で提示されたプロンプト（ハンドオフバンドルのURLを含む）を、T12実装時にそのままdev-builderに渡す。バンドルにコンポーネント構造・デザイントークン・レイアウト階層・参照アセットが含まれているため、ピクセルからの再現ではなく仕様に基づいて実装できる
- フォールバック経路: ハンドオフが使えない場合、エクスポートしたHTMLの配置先を `~/Myprogram/reading-tree/design/lp.html` とする
- 実装タスクとの対応: plan.md の T12（LPページ実装）、T5（ツリー/グラフ表示UI）
- acceptance への追記案:
  - T12: 「実装が、Claude Designのハンドオフバンドル（またはフォールバックの `design/lp.html`）のコンポーネント構造・レイアウト・コピーと一致している」
  - T5: 「実装が、Claude Designのハンドオフバンドル（またはフォールバックの `design/tree-view.html`）のグラフレイアウト・レベル別の視覚区別・ノード詳細パネルの項目と一致している」
